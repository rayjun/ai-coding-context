# 决策 #34 · Round 7 P1 §1 优先级表改名 + §6 流程规则 #2 缩短

**日期**: 2026-05-26
**状态**: Adopted

## 背景

Round 7 P0 完成跨 surface 去重后，再次审计发现两点小优化：

1. §1 推理框架里有「约束优先级」+「冲突处理优先级」两个表，名字都叫「优先级」，容易让 LLM 混淆它们是不同维度。
2. §6 流程规则 #2「目标可验证」含两个例子和一段逻辑解释，比其他 4 条规则长 4 倍。

第三项「§2 矩阵 + §6 表格合并」决定不做（影响最大章节的视觉结构，单独评估）。

## 决策

### #1 — §1 两表改名以强调维度差异

| 原名 | 新名 | 维度 |
|------|------|------|
| 约束优先级 | **决策过滤顺序** | 操作前的过滤层级 |
| 冲突处理优先级 | **取舍顺序（写代码时的冲突解决）** | 写代码时的取舍 |

两表内容不变。§3「优先级遵循 §1 取舍顺序表」同步更新。

### #2 — §6 流程规则 #2 缩短

> 原：把任务转成可检验的成功标准（例："加校验"→"为非法输入写失败测试再让它通过"）。标准越强，越能独立循环到达绿灯；标准模糊（"让它跑起来"）必然要反复澄清。
>
> 新：任务必须能转成可检验的成功标准（"让它跑起来"不算）；标准越具体，越能独立循环到绿灯。

去掉冗余例子，保留核心约束。

## 影响

实测：
- AGENTS.md：4708 → 4679 chars（**-29 字符 / ~10 tokens**）
- 节省较小因为新表名（「（写代码时的冲突解决）」）补回部分字符

正确性收益：
- §1 两表的维度差异显式化，避免 LLM 把它们当同一个「优先级」表混用
- §6 规则 #2 与其他 4 条长度趋平，规则列表视觉一致

代价：
- 极小，无功能影响

## 验证

- 旧名称（「约束优先级」/「冲突处理优先级」）grep = 0
- §3 引用 §1 「取舍顺序」表
- §6 流程规则 5 条，#2 长度与其他规则相当
